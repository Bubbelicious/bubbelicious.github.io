Demeo

Demeo is a cooperative, dungeon crawl, RPG game from Resolution Games that was originally released in VR, but was always planned to be cross-plattform with at least PC as well.
Together, up to four players will move through different "dungeons", defeating and avoiding monsters, collecting loot, and keep each other alive through a varied combination of skill-based cards.

For this game I worked on the UX design for the PC version, did research and gathered the accessibility settings that should be included in the PC version, as well as implemented some of the UI in Unity.
I also made a little surprise project for the company that is linked last of the projects on this page.

Requirements

This project was about adapting the designs from the VR game into a version that could be used on a PC or console, without losing any functions or effects. As such there were some important requirements that had to be considered that could be summarized as written here.

  • All game functions from VR version is to be included
  • Color and form must be cohesive with VR version
  • Settings are to be adapted and updated for PC and/or console usage
  • Menus are to be inviting and intuitive

You can choose one of the projects bellow for a closer look at what I worked on, but I want to point out that the project of this game has continued after I stopped working at the company, so some changes might be noticible between my work and the current look in the game.

Progression Wireframes

Progression

One of the function I worked on were making wireframes for the screen that is used for showing the player's individual progression as they level up in the game, as well as the rewards they would get for each level.
Another function is the information pane, that will be reused in places like the shop and player selection, where further information about an object or character will be needed or wanted.
We also made sure to calculate the character limits for every screen to make it easier for copywriting later.

Main Layout

Once I started working on this project, the team had already gotten some work done. The first iteration of the navigation bar didn't need my help, so the first I worked on was the main layout of the progression page, to find out what kind of size the main progression function would need and could get.
In the second iteration you can see that the information hierachy of the navigation has been improved for the users' usual prioritation. The design has also been given the official Demeo font to test the character size properly against the design.

Player Progression

Next the progression bar was messured out, thickness based on the size of surrounding icons and length based on how much white space it would need on the side to balance out the design. Above the progression bar had to contain space for reward information depending on what level the user had reached. The image for reward representation needed to be readable for the user, as well as symetrical. The amount of images was reached based on the length of the progression bar (which is why the progression bar was introduced into the design first), as well as the distance between the images that balanced out the design best.
You can see that in the second iteration, when the Demeo font and buttons were introduced, that the space for the rewards are slightly smaller. That is because the balance was somewhat thrown off once the official font was introduced and had to be compensated for.

Information pane

This information pane was originally designed to show more information about the rewards that the user could get during progression, but it was also made so that it could be adapted to any object or character in the game that needed further information.
The positive point of doing an adaption for an already existing game is that you don't have to personally make up a text that tests the wireframe design since it already exists. The design on this wireframe was generated by gathering all the information that needed to be displayed in an information pane, and find the right balance of space and text for the user to be able it skim it if they want.

Character limits

Character limits are calculated to make it easier for fonetic copywriting. Using the letter "W" makes sure that any kind of words will fit within the limit since it is one of the broadest letters no matter the font that is used in the end. This way, once the team starts with copywriting, they know how many character that will fit, as well as what font size they will need to use so that the users' can read it easily.

Play Action Wireframes

Play Actions

One of the functions that I worked on were making wireframes for the screens that are used when the player choses a Play action. The actions in question are: Join Game or Create Game, and the designs had to include a screen for joining a specific room, loading screens, and error screens.
We also made sure to calculate the character limits for every screen to make it easier for copywriting later.

Join Game

If a player want to join a specific game, with friends for example, they would need to enter the correct room number that the creator of the game session have to share with them. Below you can see my first and second iteration of the screen to choose which game room the player want to join.
Since the actions and information that had to be included into the page such a small amount, and don't take up much room on the page, I decided that a centered design made more sense. The left over space around can easily be filled in with mood-setting illustrations or similar, and a centered design would be easier to balance for the UI artist.

A loading screen is essential for all games to make sure that the player both know that the game is working on their request, as well as helping the player stay in the mood of the game while it loads. This page can also be adapted to fit other loading interactions by changing out information text and background illustrations. These are the first and second iteration of the Loading screen when player has chosen the game room they want o join, as well as an example on how it can be adapted for a player that create their own game session.

You never want something in the game to fail, but they do. Sometimes it's because of creation error, sometimes a connection error, and sometimes it's because of player error. Either way, there need to be a page in the game that tells the player what has gone wrong, and give them options on how it can be fixed. Error messages are an important part in making the player feel secure, and should always be included in the design process. Below you can see iterations of error screens for when the chosen game room failed to load for some reason, as well as if the room could not be created for a new session. It can also be adapted into other parts of the game by changing the information and options available.

UI in Unity

Unity

Unity is a game engine that is used as a tool that allows someone to do all kinds of work related to game development. For this project we, the UX designers, are working in the 2D platform to create menus for a 3D game play. Once the UI designer finishes the look of the menus, the programmers will add them into the game with the UI designer's help. My contribution to this part of the project was importing the components from the wireframes into Unity so that they can be tested and further developed.

In Game Menu

Toning out the gameplay, the In Game menu gives the player an opportunity to adjust the settings, check their inventory and/or make changes to their character, read up on the game mechanics, etc, without having to end their game session.
The button design was imported and multiplied before they were adapted for their specific purpose, to make sure that later iterations kept to the right sizes, and the menu's background was added from the original VR game's UI library that was already available.

The Inventory button in the In Game menu was coded to bring the player to their Inventory page. In this page, the grid of category images were imported in rows before they were adapted for their specific purpose, to make sure that later iterations kept to the right sizes. By this point in the project we had no illustrations that could be used since all earlier illustrations were for a VR perspective, so the categories had to make do with names only. When this page was later tested, they found that without illustrations they couldn't test it properly because the test-user didn't understand the different categories.

The Progression button in the In Game menu was coded to bring the player to their Progression page. In this page, the reward images were imported as a row and was then coded into a carusel of three images at the time, with the buttons on either side to move it, to make sure that later iterations kept to the right sizes. The progression bar was not imported for this step because the testing of the rewards were the focus at the time.

The Settings button in the In Game menu was coded to bring the player to their Setting page. In this page, the buttons for the different categories were imported and then adapted and coded for their specific purpose, to make sure that later iterations kept to the right sizes. In the different settings there were functional and coded sliders, drop-downs, or click options for every setting opportunity. The list of which settings can be found in Setting Selection further down on this page.

The Help button in the In Game menu was coded to bring the player to the Help page. In this page, the buttons for the different options were imported and then adapted and coded for their specific purpose, to make sure that later iterations kept to the right sizes. They were designed in such a way that you could add more information if/when more gameplay was added. No actual help information was added in this iteration, since the testing that was planned was only targeting navigation and not information.

Character Selection

In the Character Selection page, the player could choose which character they wanted to play as. The player could take a look at the different characters in a 3D perspective, and if they had it they might choose to change the skin of the character before selecting it. The images of the characters were coded to light up the character that was currently being looked at, but since we didn't had 3D images of all the characters available, the front character didn't change.
When this iteration of the Character Selection page was made, there were only four different characters that could be chosen from.

In Game Shop

In the In Game Shop, which can be found between levels in the gameplay, the player can buy new cards for their hand with the coin they might have collected during the game, as well as sell cards from their hand that they have no use for. The important part of this design was that the shady trader character (whose name changed between iterations) had to be visible in some manner so that it still had the same In Game feeling as the VR version. Buying or Selling cards depended on the chosen category over the grid of cards, and you can see which players that have finished their shopping in the row of characters underneath the card grid. The card shapes were imported to make sure that the illustrations of later iterations kept to the proper sizes.
